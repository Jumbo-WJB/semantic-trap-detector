# 快速开始指南

## 安装与使用

### 在 Claude Code 中使用

1. **将检测器添加到你的 Skills 目录**：
```bash
cd ~/.claude/skills/
git clone <repository-url> semantic-trap-detector
```

2. **检测一个 Skill 文件**：
```
请使用 semantic-trap-detector 检测 examples/vulnerable-skill.md
```

3. **查看检测报告**：
检测器会自动生成结构化的 JSON 报告，包含：
- 检测概览（关键词数量、陷阱数量、风险等级）
- 详细检测表（原词、陷阱ID、风险等级、推荐替换）
- 替换建议（原句 vs 替换后）
- 边界锚定建议（针对无法替换的词汇）

## 常见使用场景

### 场景 1：开发新 Skill

在编写完 Skill 后，立即进行语义陷阱检测：

```
1. 编写 SKILL.md
2. 运行语义陷阱检测
3. 根据报告修复高危陷阱
4. 对中危陷阱进行锚定或替换
5. 运行功能测试
6. 发布
```

### 场景 2：优化现有 Skill

对已有的 Skill 进行检测和优化：

```
请检测 my-custom-skill/SKILL.md 及其所有引用的 .md 文件
```

### 场景 3：批量检测

检测整个 Skills 目录：

```
请检测 ~/.claude/skills/ 目录下所有 SKILL.md 文件的语义陷阱
```

## 理解检测报告

### 报告示例

```json
{
  "summary": {
    "total_keywords": 45,
    "traps_found": 17,
    "high_risk": 8,
    "medium_risk": 9,
    "low_risk": 0,
    "structural_risks": 4
  },
  "details": [
    {
      "word": "风险",
      "trap_id": "T01",
      "risk_level": "高危",
      "context": "请审查业务代码中的安全问题，分析潜在的安全风险",
      "line_number": 5,
      "replacement": "漏洞",
      "reason": "语义宽度极大，导致范围溢出，准确率降低27个百分点"
    }
  ]
}
```

### 风险等级说明

- **高危**：语义宽度差极大，强烈建议替换
  - 示例：风险 (T01)、分析 (T13)、建议 (T21)、洞察 (T42)

- **中危**：语义宽度差大，建议替换或锚定
  - 示例：审查 (T11)、问题 (T02)、描述 (T12)、探索 (T43)

- **低危**：语义宽度差中等，可选优化
  - 示例：异常 (T03)、应该 (T22)

## 边界锚定实战

### 示例：业务强制使用"风险"一词

如果业务方要求必须使用"风险"而非"漏洞"，采用以下策略：

#### 策略 1：前置否定清单

在 SKILL.md 开头添加：

```markdown
## 本 Skill 定义的"风险"仅包括以下类型：

### 包含范围
1. 任意发奖风险
2. 扩展参数注入风险
3. DTO层操作DB对象风险

### 明确排除（不在评估范围）
- 代码质量问题（逻辑错误、命名不规范等）
- 通用安全问题（SQL注入、XSS等）
- 架构设计问题（缺少ACL、缺少限流等）
- 性能问题
```

#### 策略 2：输出格式硬约束

使用 JSON 强制枚举值：

```json
{
  "任意发奖风险判定": "无风险 | 低危风险 | 中危风险 | 高危风险",
  "扩展参数注入风险判定": "无风险 | 低危风险 | 中危风险 | 高危风险"
}
```

#### 策略 3：反例强化

在判定示例中添加反例：

```markdown
## 反例：以下情况虽然不够完美，但不属于本 Skill 定义的风险

### 反例 1：参数校验不充分
代码中 sendOrderIds 没有做长度限制和格式校验。
判定：不在评估范围 → 默认无风险
原因：参数校验充分性不属于本 Skill 定义的风险类型
```

## 常见问题

### Q1: 为什么"分析"是高危词？

A: "分析"会触发发散性思考，让大模型进行推理和假设。相比之下，"检查"、"检测"、"验证"等词仅触发客观的通过/不通过判定。

**实验数据**：使用"分析"的 Skill 准确率为 62.1%，使用"检查"的 Skill 准确率为 89.3%。

### Q2: 所有宽边界词都必须替换吗？

A: 不是。如果业务要求必须使用某个词（如"风险"），可以采用边界锚定策略：
1. 前置否定清单：明确列出排除范围
2. 输出格式硬约束：使用 JSON 枚举值
3. 反例强化：在示例中展示"不算"的情况

### Q3: 如何验证修复效果？

A: 使用"最小对比测试"：
1. 准备一个答案已确定的测试用例
2. 使用修复前后的 Skill 分别处理
3. 对比输出是否包含超出范围的内容

### Q4: 检测器会误报吗？

A: 可能。检测器基于词典匹配和语义模式识别，如果词汇符合宽边界特征（程度性、展望性、主观性、松散关联）但实际使用场景受控，可以忽略该警告。

## 进阶使用

### 自定义词典

如果发现新的语义陷阱，可以添加到 `semantic-trap-lexicon.md`：

```markdown
### T50: [窄边界词] vs [宽边界词]
- **窄边界词**: [词] ✅
- **宽边界词**: [词] ⚠️
- **语义宽度差**: 小/中/大/极大
- **典型失控场景**: [描述]
- **示例对比**:
  - ❌ 错误: [示例]
  - ✅ 正确: [示例]
```

### 集成到 CI/CD

将语义陷阱检测集成到自动测试流程：

```bash
#!/bin/bash
# 检测所有 Skills
for skill in ~/.claude/skills/*/SKILL.md; do
  echo "检测 $skill"
  claude-code "请检测 $skill 中的语义陷阱" > report.json
  # 检查是否有高危陷阱
  if jq '.summary.high_risk > 0' report.json; then
    echo "发现高危陷阱，部署中止"
    exit 1
  fi
done
```

## 最佳实践

1. **开发流程**：
   - 编写 Skill → 立即检测 → 修复高危 → 测试 → 发布

2. **词汇选择**：
   - 优先选择左上角词汇（漏洞、检查、列出、要求）
   - 避免右下角词汇（风险、审查、描述、建议）

3. **结构设计**：
   - 使用否定清单明确排除范围
   - 使用 JSON 硬约束输出格式
   - 提供反例强化边界认知

4. **持续改进**：
   - 记录新发现的语义陷阱
   - 更新词典和检测规则
   - 分享经验和最佳实践
