# 语义陷阱检测器 (Semantic Trap Detector)

一个专门用于检测和优化 Claude Skills 的元 Skill，基于**语义陷阱理论**自动识别高风险词汇和结构模式。

## 🎯 核心功能

- ✅ **自动检测**：系统化扫描 Skill 文件中的语义陷阱
- 📊 **风险分级**：高/中/低三级风险评估
- 🔄 **智能替换**：提供具体的词汇替换建议
- 🛡️ **边界锚定**：针对无法替换的词汇提供约束策略
- 📝 **结构化报告**：生成 JSON 格式的可执行报告

## 📖 什么是语义陷阱？

**语义陷阱**（Semantic Trap）：在人类日常语境中含义相近，但在大模型的语义空间中激活范围（语义边界）存在巨大差异的词汇对。

### 一个惊人的数据

| Skill | 核心术语 | 正确率 |
|-------|---------|--------|
| biz-vul-security | 漏洞 | **89.3%** |
| biz-risk-security | 风险 | **62.1%** |

**27个百分点**的差异仅源于一个词的替换！

```bash
# 检测单个 Skill 文件

"请检测 SKILL.md 并提供替换建议和锚定方案"
```

## 🧪 检测示例

### 示例 1：核心术语陷阱

**原文**：
```
请审查这段代码中的安全问题，只关注定义文件中的问题类型。
```

**检测结果**：
- "审查" (T11, 中危) → 触发主观评价
- "问题" (T02, 高危) → 语义范围极宽

**替换后**：
```
请检查这段代码中的安全漏洞，只关注定义文件中的漏洞类型。
```

### 示例 2：动作指令陷阱

**原文**：
```
分析代码质量，给出优化建议
```

**检测结果**：
- "分析" (T13, 高危) → 触发散发性思考
- "建议" (T21, 高危) → 大量创意性输出

**替换后**：
```
检查代码规范遵循情况，列出必须满足的要求
```

```
            高 ←————→ 低
        ┌───────────────────┐
 低    │  漏洞  缺陷  错误  │  ← 安全区 ✅
 ↑     │  检查  列出  要求  │
 发    │─────────────────── │
 散    │  总结  问题  审查  │
 倾    │  建议  分析  风险  │  ← 危险区 ⚠️
 向    │  异常  评估  描述  │
 ↓     └───────────────────┘
 高
```

## 🛡️ 边界锚定策略

当业务要求必须使用宽边界词时，可采用以下策略：

### 策略 1：前置否定清单

```markdown
## 本 Skill 定义的"风险"仅包括：
1. 任意发奖风险
2. 扩展参数注入风险

## 以下不属于评估范围：
- 代码质量问题（逻辑错误、命名不规范等）
- 通用安全问题（SQL注入、XSS等）
- 架构设计问题（缺少ACL、缺少限流等）
```

### 策略 2：输出格式硬约束

```json
{
  "任意发奖风险判定": "无风险 | 低危风险 | 中危风险 | 高危风险",
  "扩展参数注入风险判定": "无风险 | 低危风险 | 中危风险 | 高危风险"
}
```

### 策略 3：反例强化

```markdown
## 反例：以下情况虽然不够完美，但不属于本 Skill 定义的风险

### 反例 1：参数校验不充分
代码中 sendOrderIds 没有做长度限制和格式校验。
判定：不在评估范围 → 默认无风险
原因：参数校验充分性不属于本 Skill 定义的风险类型
```

## 📚 词典内容

当前收录 **17 组中文词汇对** + **10 组英文词汇对**，覆盖 4 大类别：

### 核心术语类 (T01-T06)
- 漏洞 vs 风险
- 缺陷 vs 问题
- 错误 vs 异常

### 动作指令类 (T11-T16)
- 检查 vs 审查
- 列出 vs 描述
- 总结 vs 分析

### 约束词类 (T21-T24)
- 要求 vs 建议
- 必须 vs 应该

### 输出描述类 (T31-T33)
- 列表 vs 报告

详细内容请查看 `semantic-trap-lexicon.md`

## 🔍 检测工作流

本检测器采用严格的四步工作流：

```
STEP 1: 加载词典
   ↓
STEP 2: 提取关键词（核心术语、动作指令、约束词、输出描述词）
   ↓
STEP 3: 逐词匹配与上下文分析（两层判定）
   ↓
STEP 4: 生成检测报告
```

## ⚙️ 使用建议

### 开发流程集成

建议将语义陷阱检测作为 Skill 开发的标准环节：

```
Skill 编写 → 语义陷阱检测 → 修复/锚定 → 功能测试 → 发布
```

类似于代码开发中的静态分析（Lint）。

### 优先级处理

1. **立即修复**：所有高危陷阱（T01, T13, T21）
2. **尽快修复**：中危陷阱（T11, T02, T12）
3. **可选优化**：低危陷阱和疑似陷阱

## 🧪 验证方法

### 最小对比测试

1. 准备一个答案已确定的测试用例
2. 使用当前 Skill 进行处理
3. 检查输出是否包含超出范围的内容

### 排除测试

将约束语句提供给大模型：
```
"除了定义文件中明确列出的类型，还有哪些 X 你认为也应当纳入关注范围？"
```

如果能列举出大量额外内容，说明核心术语语义边界过于宽泛。

## 📖 理论来源

本检测器基于以下研究成果：

- **论文**：《别让大模型"想太多"：SKILL开发中的语义陷阱与抗幻觉设计》
- **作者**：SummerSec
- **实验**：56个营销接口测试集，27个百分点的准确率差异
- **核心发现**：语义边界控制比传统 Prompt Engineering 更为根本

## 💡 核心洞察

> 选择了语义边界精准的词汇，你的 Skill 约束体系就如同一个高效的漏斗。
> 选择了语义边界过宽的词汇，再精良的工作流也会被从内部撑破。

**在大下次编写 Skill 时，先停下来审视：这些关键词汇在大模型的认知世界里，真的和你以为的一样大吗？**
