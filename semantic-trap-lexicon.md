# 语义陷阱词典 (Semantic Trap Lexicon)

## 词典说明

本词典收录了在Skill开发中容易导致大模型幻觉的词汇对。每对词汇在人类日常语境中含义相近，但在大模型的语义空间中激活范围（语义边界）存在巨大差异。

使用语义边界过宽的词汇，会导致大模型的输出突破Prompt所设定的约束范围。

---

## LLM 语义敏感度矩阵

```
            边界清晰度
            高 ←————→ 低
        ┌───────────────────┐
 低    │  漏洞  缺陷  错误  │  ← 安全区
 ↑     │  检查  列出  要求  │
 发    │─────────────────── │
 散    │  总结  问题  审查  │
 倾    │  建议  分析  风险  │  ← 危险区
 向    │  异常  评估  描述  │
 ↓     └───────────────────┘
 高
```

矩阵中越靠近左上角的词汇，在大模型中的语义表征越收敛，对应的输出也越具可控性。越靠近右下角的词汇，语义越趋于发散，大模型进行"自由创作"的空间越大。

## 核心术语类陷阱 (T01-T06)

### T01: 漏洞 vs 风险
- **窄边界词**: 漏洞 (Vulnerability) ✅
- **宽边界词**: 风险 (Risk) ⚠️
- **语义宽度差**: 极大
- **典型失控场景**:
  - "风险"会触发发散性思考，将代码质量、架构设计等问题纳入报告
  - "漏洞"具有明确的二元判定标准（存在/不存在），语义边界清晰
- **示例对比**:
  - ❌ 错误: "评估代码安全风险" → 输出代码逻辑错误、参数校验不足等
  - ✅ 正确: "检测安全漏洞" → 仅输出符合漏洞定义的具体缺陷

### T02: 缺陷 vs 问题
- **窄边界词**: 缺陷 (Defect) ✅
- **宽边界词**: 问题 (Issue) ⚠️
- **语义宽度差**: 大
- **典型失控场景**:
  - "问题"的语义范围极宽，改进建议、优化点都可能被归类为"问题"
  - "缺陷"指向明确的、可复现的错误
- **示例对比**:
  - ❌ 错误: "列出代码中的问题" → 输出命名不规范、性能优化建议等
  - ✅ 正确: "识别代码缺陷" → 仅输出功能性缺陷

### T03: 错误 vs 异常
- **窄边界词**: 错误 (Error) ✅
- **宽边界词**: 异常 (Anomaly) ⚠️
- **语义宽度差**: 中等
- **典型失控场景**:
  - "异常"的边界定义模糊，轻微偏差都可能被纳入报告
  - "错误"通常有明确的异常类型和错误信息

## 动作指令类陷阱 (T11-T16)

### T11: 检查 vs 审查
- **窄边界词**: 检查 (Check) ✅
- **宽边界词**: 审查 (Review) ⚠️
- **语义宽度差**: 大
- **典型失控场景**:
  - "审查"会触发主观评价性意见
  - "检查"仅触发通过/不通过的客观判定
- **示例对比**:
  - ❌ 错误: "审查代码质量" → 输出主观评价和建议
  - ✅ 正确: "检查是否遵循规范" → 输出是/否判定

### T12: 列出 vs 描述
- **窄边界词**: 列出 (List) ✅
- **宽边界词**: 描述 (Describe) ⚠️
- **语义宽度差**: 大
- **典型失控场景**:
  - "描述"会触发解释和评论性内容
  - "列出"仅产出结构化条目
- **示例对比**:
  - ❌ 错误: "描述发现的问题" → 输出详细分析和评论
  - ✅ 正确: "列出所有发现" → 输出简洁的列表

### T13: 总结 vs 分析
- **窄边界词**: 总结 (Summarize) ✅
- **宽边界词**: 分析 (Analyze) ⚠️
- **语义宽度差**: 很大
- **典型失控场景**:
  - "分析"会触发推理和假设性思考
  - "总结"仅提炼已有信息
- **示例对比**:
  - ❌ 错误: "分析用户行为" → 输出推理和假设
  - ✅ 正确: "总结用户反馈" → 输出提炼的要点

## 约束词类陷阱 (T21-T24)

### T21: 要求 vs 建议
- **窄边界词**: 要求 (Requirement) ✅
- **宽边界词**: 建议 (Suggestion) ⚠️
- **语义宽度差**: 很大
- **典型失控场景**:
  - "建议"会触发发散性思维，产生大量创意性内容
  - "要求"仅输出明确条件
- **示例对比**:
  - ❌ 错误: "给出优化建议" → 输出大量创新方案
  - ✅ 正确: "列出必须满足的要求" → 输出明确的条件列表

### T22: 必须 vs 应该
- **窄边界词**: 必须 (Must) ✅
- **宽边界词**: 应该 (Should) ⚠️
- **语义宽度差**: 中等
- **典型失控场景**:
  - "应该"约束力较弱，大模型可能选择性遵守
  - "必须"具有强制性约束力

---

## 输出描述类陷阱 (T31-T33)

### T31: 列表 vs 报告
- **窄边界词**: 列表 (List) ✅
- **宽边界词**: 报告 (Report) ⚠️
- **语义宽度差**: 中等
- **典型失控场景**:
  - "报告"会触发结构化文档，包含引言、分析、建议等
  - "列表"仅输出条目化内容

## 扩展词汇对 (基于语义模式推断)

### T41: 评估 (Evaluate) ⚠️
- **风险等级**: 高
- **特征**: 程度性、主观性、展望性
- **替换建议**: 测试 (Test)、验证 (Validate)、判定 (Determine)

### T42: 洞察 (Insight) ⚠️
- **风险等级**: 高
- **特征**: 高度发散、触发创造性思考
- **替换建议**: 发现 (Findings)、结果 (Results)、数据 (Data)

### T43: 探索 (Explore) ⚠️
- **风险等级**: 中
- **特征**: 开放式、无明确边界
- **替换建议**: 搜索 (Search)、查找 (Find)、定位 (Locate)

### T44: 理解 (Understand) ⚠️
- **风险等级**: 中
- **特征**: 难以验证、主观性强
- **替换建议**: 识别 (Identify)、提取 (Extract)、解析 (Parse)

## 结构性风险模式

除了单个词汇的陷阱，以下结构模式也会导致语义发散：

### 模式1: 开放式动词 + 无范围限定
- **示例**: "分析代码"（未指定分析维度）
- **风险**: 大模型自行决定分析方向
- **修复**: 明确分析范围，如"分析代码中的安全漏洞"

### 模式2: 形容词化目标
- **示例**: "评估安全性"而非"检测安全漏洞"
- **风险**: 抽象名词扩大范围
- **修复**: 使用具体名词

### 模式3: 情态动词降级
- **示例**: 约束条件用"应该"而非"必须"
- **风险**: 约束力削弱
- **修复**: 使用强制性词汇

### 模式4: 缺少否定清单
- **示例**: 使用宽边界词但未列出排除项
- **风险**: 无边界兜底
- **修复**: 添加明确的排除列表

## 边界锚定策略

当业务要求必须使用宽边界词时，可采用以下策略：

### 策略1: 前置否定清单
在定义文件最前面添加明确的"不包括"列表：
```markdown
## 本 Skill 定义的"风险"仅包括以下类型：
1. 任意发奖风险
2. 扩展参数注入风险

## 以下不属于本 Skill 评估范围（即使它们在通常意义上也是"风险"）：
- 代码质量问题（逻辑错误、命名不规范等）
- 通用安全问题（SQL注入、XSS等）
- 架构设计问题（缺少ACL、缺少限流等）
```

### 策略2: 输出格式硬约束
将输出格式设计为结构化的"填空题"：
```json
{
  "任意发奖风险判定": "无风险 | 低危风险 | 中危风险 | 高危风险",
  "扩展参数注入风险判定": "无风险 | 低危风险 | 中危风险 | 高危风险"
}
```

### 策略3: 判定示例的反例强化
在Few-shot示例中增加反例：
```markdown
## 反例：以下情况虽然不够完美，但不属于本 Skill 定义的风险

### 反例 1：参数校验不充分
代码中 sendOrderIds 没有做长度限制和格式校验。
判定：不在评估范围 → 默认无风险
原因：参数校验充分性不属于本 Skill 定义的风险类型
```

## 词典使用指南

1. **关键词提取时**: 将待检测文本中的词汇与本词典对照
2. **风险评估时**: 根据陷阱ID和风险等级确定处理优先级
3. **替换建议时**: 优先选择同类别中的窄边界词
4. **锚定策略时**: 当无法替换时，应用边界锚定策略
